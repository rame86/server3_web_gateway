MSA의 핵심 지향점: 분리와 자율성
프로젝트 지향성
 1. MSA의 핵심 지향점: 분리와 자율성
MSA는 서비스들을 독립적으로 배포하고 확장할 수 있는 상태를 만드는 것이 목표.
Bounded Context (제한된 맥락): 각 서비스는 비즈니스의 특정 기능(도메인)에만 집중해야 합니다. 서비스 간 데이터베이스 공유는 금기이며, 필요시 API를 통해서만 데이터를 주고받음.
Service Autonomy (서비스 자율성): 특정 서비스에 장애가 발생해도 전체 시스템이 중단되지 않아야 합니다(내결함성). 또한, 서비스별로 최적의 기술 스택(Polyglot)을 선택할 수 있어함.
Decentralized Data Management: 데이터는 각 서비스가 소유합니다. 트랜잭션 관리는 2PC(Two-Phase Commit)보다는 Saga 패턴 등을 통한 결과적 일관성(Eventual Consistency)을 지향.
2. RESTful API 제약 조건
RESTful 원칙
원칙 (Principle)
설명 (Description)
Uniform Interface
리소스(URI)가 식별되어야 하며, HTTP 메서드(GET, POST 등)를 일관되게 사용
Stateless (무상태성)
서버는 클라이언트의 상태를 저장하지 않습니다. 각 요청은 처리에 필요한 모든 정보를 포함
Cacheable (캐시 가능성)
HTTP의 표준 캐싱 기능을 사용하여 응답 데이터의 재사용성이 높아야 함
Client-Server 구조
클라이언트(UI)와 서버(데이터/비즈니스 로직)가 서로 독립적으로 진화
Layered System
클라이언트는 서버가 중간 매체(게이트웨이)를 거치는지 직접 통신하는지 몰라야 함
Code on Demand (옵션)
서버에서 스크립트를 클라이언트로 보내 실행할 수 있게 함

3. 보안 (Security)
보안 및 인증 시스템 아키텍쳐
Server 3(Gateway)가 진입점 역할을 수행 Server 1(Core & Redis)과 통신하여 인증을 처리하는 중앙 집중형 보안구조
인증 방식: JWT 및 Redis 기반 중앙 집중형 제어
인증 정보의 일관성을 유지하고 중복 로그인을 제어하기 위해 JWT와 Redis를 결합한 방식을 사용합니다.
기본 인증 수단: JWT (JSON Web Token)를 사용하여 클라이언트에게 무상태(Stateless) 인증 수단을 제공.
세션 및 상태 관리: Server 1의 Redis(34.158.208.117:6379)를 활용하여 JWT 인증 정보와 유저 데이터를 관리.
토큰 발급 주체: Server 1의 Core Service(/msa/core/)가 회원가입 및 로그인 처리를 전담하며 유효한 토큰을 발급.
API Gateway (OpenResty + Lua) 보안 파이프라인
Server 3에 구축된 OpenResty(Nginx + Lua)가 모든 외부 요청의 보안 필터 역할을 수행

서비스별 라우팅 및 접근 제어
서비스의 성격에 따라 게이트웨이 통과 시 인증 적용 여부를 구분
A. 인증 예외 구간 (Public Access)
로그인하지 않은 사용자도 접근 가능한 서비스로, Gateway의 auth.lua 검증 과정을 거치지 않음
대상 서비스: Core Service
주요 경로: /msa/core/ (회원가입, 로그인 등)
설정 특징: access_by_lua_file 설정이 제외됨
B. 인증 필수 구간 (Protected Access)
반드시 유효한 토큰이 있어야만 접근 가능한 도메인 서비스
대상 서비스: Payment, Shop, Reservation Service
주요 경로: /msa/pay/, /msa/shop/, /msa/res/
보안 적용: 모든 요청은 프록시 전달 전 auth.lua를 통해 Redis 인증을 통과해야함

개발환경 구축
MSA 개발환경 구축
모든 서비스가 분산되어 있는 시스템상 여러 게이트웨이와 메세지큐, 레디스같은 임시 저장소가 필요한 개발환경에서
개발자 모두가 동일한 개발환경을 구성하기위해 docker-compose를 이용한 local 컨테이너에서 개발을 진행함
개발 인프라 구축 파일 구조도
develop_container/  ├── nginx/  │   ├── conf.d/  │   │    └── default.conf     # Nginx 서버 설정 파일  │   └── lua/  │        └── auth.lua         # Lua를 이용한 인증 로직 스크립트  ├── .env                      # 환경 변수 설정 파일  └── docker-compose.yml        # 컨테이너 오케스트레이션 설정
환경 변수 (.env)
팀원들이 공통으로 사용할 로컬 환경 변수
# develop_container/.env
# RabbitMQ
RABBITMQ_USER=msa_mq
RABBITMQ_PASSWORD=mq1234
로컬 인프라 통합 (docker-compose.yml)
동일한 로컬 인프라를 구축 및 별도의 bridge 네트워크를 지정하여 개발환경 최적화.
서비스 디스커버리 (Service Discovery):
네트워크 내에 있는 컨테이너들은 서로의컨테이너 이름(Service Name)을 도메인 이름처럼 사용하여 통신
예를 들어, redis-commander 설정에서 IP 주소 대신 local-redis:6379라고 적어도 Docker 내부 DNS가 자동으로 해당 컨테이너의 IP로 연결. 컨테이너가 재시작되어 IP가 바뀌어도 설정을 변경할 필요가 없음.
네트워크 격리 및 보안 (Isolation):
기본(default) 네트워크를 사용하지 않고 전용 네트워크를 사용하면, 현재 프로젝트와 무관한 다른 Docker 컨테이너들로부터 인프라를 논리적으로 분리.
개발 환경이 복잡해질수록 다른 프로젝트의 컨테이너와 이름이 겹치거나 포트가 충돌하는 문제를 방지.
환경 일관성:
모든 팀원이 동일한 docker-compose.yml을 실행하면, 각자의 로컬 PC 환경에 상관없이 컨테이너 간의 통신 경로가 동일하게 유지.

      - "6379:6379"

      - local-msa-network # 내부 서비스 간 통신용 네트워크

Lua 인증 스크립트 (nginx/lua/auth.lua)
Docker 네트워크 내의 Redis 컨테이너 이름(local-redis)으로 접속하도록 설정
-- develop_container/nginx/lua/auth.lua
-- [모듈 로드] Redis 클라이언트 및 JSON 파서 로드
local redis = require "resty.redis"
local cjson = require "cjson.safe"
local red = redis:new()
-- [설정] Redis 연결 타임아웃 설정 (Connect, Send, Read 순서 / 단위: ms)
red:set_timeouts(1000, 1000, 1000)
-- [연결] Redis 서버 연결 시도 (컨테이너명 혹은 IP, 포트)
local ok, err = red:connect("local-redis", 6379)
if not ok then
    ngx.log(ngx.ERR, "Redis 연결 실패: ", err)
    ngx.exit(ngx.HTTP_INTERNAL_SERVER_ERROR)
end
-- =========================================================================
-- 1. Authorization 헤더 검증 및 Bearer 토큰 추출
-- =========================================================================
local auth_header = ngx.var.http_authorization
if not auth_header then
    ngx.log(ngx.WARN, "Authorization 헤더가 없습니다.")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- Bearer [Token] 형식인지 정규식으로 검사 및 토큰 본문 캡처
local token = ngx.re.match(auth_header, "Bearer\\s+(.+)", "jo")
if not token then
    ngx.log(ngx.ERR, "토큰 형식이 유효하지 않음: ", auth_header)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
local pure_token = token[1]
-- =========================================================================
-- 2. JWT 로컬 파싱 (Redis 조회 전 Key 생성을 위한 member_id 추출)
-- 별도의 라이브러리 없이 JWT 구조(Header.Payload.Signature) 중 Payload를 디코딩
-- =========================================================================
-- 정규식을 이용해 첫 번째 점(.)과 두 번째 점 사이의 Payload 영역 추출
local payload_b64url = ngx.re.match(pure_token, "^[^%.]+\\.([^%.]+)\\.", "jo")
if not payload_b64url then
    ngx.log(ngx.ERR, "JWT 형식이 올바르지 않음")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- Base64Url 형식을 표준 Base64 형식으로 변환 (특수문자 치환 및 패딩 추가)
local b64 = payload_b64url[1]:gsub("-", "+"):gsub("_", "/")
local remainder = #b64 % 4
if remainder > 0 then
    b64 = b64 .. string.rep("=", 4 - remainder)
end
-- Base64 디코딩 수행 및 JSON 역직렬화
local payload_json = ngx.decode_base64(b64)
local payload_data = cjson.decode(payload_json)
-- Payload 내 식별자 존재 여부 확인 (표준 클레임인 sub 사용)
if not payload_data or not payload_data.sub then
    ngx.log(ngx.ERR, "토큰 Payload에서 sub(member_id)를 찾을 수 없음")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- sub 값을 member_id 변수로 매핑
local member_id = payload_data.sub
-- =========================================================================
-- 3. Redis 세션 검증 (Hash 타입 구조 반영)
-- member_id를 기반으로 저장된 최신 세션 정보를 Hash 형태로 조회
-- =========================================================================
local redis_key = "AUTH:MEMBER:" .. tostring(member_id)
-- Hash 전용 명령어 hgetall 사용
local res, err = red:hgetall(redis_key)
if err then
    ngx.log(ngx.ERR, "Redis 조회 에러: ", err)
    ngx.exit(ngx.HTTP_INTERNAL_SERVER_ERROR)
end
-- hgetall은 키가 존재하지 않으면 빈 테이블({})을 반환함
if not res or #res == 0 then
    ngx.log(ngx.ERR, "로그아웃 되었거나 세션이 만료된 사용자: ", redis_key)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- 평면 배열 형식([ "token", "ey...", "role", "ARTIST" ])을 Lua Key-Value 테이블로 변환
local session_data = {}
for i = 1, #res, 2 do
    session_data[res[i]] = res[i + 1]
end
-- =========================================================================
-- 4. 보안 정책 검증 (Token Consistency Check)
-- 클라이언트가 보낸 토큰과 Redis에 저장된 '현재 유효한 토큰'이 일치하는지 비교
-- =========================================================================
if session_data.token ~= pure_token then
    ngx.log(ngx.ERR, "토큰 불일치 (중복 로그인 등에 의해 갱신됨): ", member_id)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- =========================================================================
-- 5. 게이트웨이 통과 및 헤더 주입
-- 인증된 정보를 Downstream(마이크로서비스) 헤더에 담아 전송
-- =========================================================================
ngx.req.clear_header("X-User-Id")
ngx.req.clear_header("X-Role")
ngx.req.set_header("X-User-Id", tostring(member_id))
ngx.req.set_header("X-Role", tostring(session_data.role))
-- Redis 커넥션 풀 반환
red:set_keepalive(10000, 100)

-- /home/suyeong1526/nginx/lua/auth.lua
-- [모듈 로드] Redis 클라이언트 및 JSON 파서 로드
local redis = require "resty.redis"
local cjson = require "cjson.safe"
local red = redis:new()
-- [설정] Redis 연결 타임아웃 설정 (Connect, Send, Read 순서 / 단위: ms)
red:set_timeouts(1000, 1000, 1000)
-- [연결] Redis 서버 연결 시도 (컨테이너명 혹은 IP, 포트)
local ok, err = red:connect("local-redis", 6379)
if not ok then
    ngx.log(ngx.ERR, "Redis 연결 실패: ", err)
    ngx.exit(ngx.HTTP_INTERNAL_SERVER_ERROR)
end
-- =========================================================================
-- 1. Authorization 헤더 검증 및 Bearer 토큰 추출
-- =========================================================================
local auth_header = ngx.var.http_authorization
if not auth_header then
    ngx.log(ngx.WARN, "Authorization 헤더가 없습니다.")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- Bearer [Token] 형식 정규식 검사 및 토큰 본문 캡처
local token = ngx.re.match(auth_header, "Bearer\\s+(.+)", "jo")
if not token then
    ngx.log(ngx.ERR, "토큰 형식이 유효하지 않음: ", auth_header)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
local pure_token = token[1]
-- =========================================================================
-- 2. JWT 로컬 파싱 (Redis 조회 전 Key 생성을 위한 member_id 추출)
-- Payload 영역을 추출하여 Base64Url 디코딩 수행
-- =========================================================================
-- JWT 구조(Header.Payload.Signature) 중 Payload 영역 추출
local payload_b64url = ngx.re.match(pure_token, "^[^%.]+\\.([^%.]+)\\.", "jo")
if not payload_b64url then
    ngx.log(ngx.ERR, "JWT 형식이 올바르지 않음")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- Base64Url -> 표준 Base64 변환 및 패딩 추가
local b64 = payload_b64url[1]:gsub("-", "+"):gsub("_", "/")
local remainder = #b64 % 4
if remainder > 0 then
    b64 = b64 .. string.rep("=", 4 - remainder)
end
-- Base64 디코딩 및 JSON 역직렬화
local payload_json = ngx.decode_base64(b64)
local payload_data = cjson.decode(payload_json)
-- Payload 내 식별자(sub) 존재 확인
if not payload_data or not payload_data.sub then
    ngx.log(ngx.ERR, "토큰 Payload에서 sub(member_id)를 찾을 수 없음")
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
local member_id = payload_data.sub
-- =========================================================================
-- 3. Redis 세션 검증 (Hash 타입 구조 반영)
-- member_id 기반으로 저장된 세션 정보를 Hash 형태로 조회
-- =========================================================================
local redis_key = "AUTH:MEMBER:" .. tostring(member_id)
-- Hash 전용 명령어 hgetall 사용
local res, err = red:hgetall(redis_key)
if err then
    ngx.log(ngx.ERR, "Redis 조회 에러: ", err)
    ngx.exit(ngx.HTTP_INTERNAL_SERVER_ERROR)
end
-- 키 미존재 시 401 반환
if not res or #res == 0 then
    ngx.log(ngx.ERR, "로그아웃 되었거나 세션이 만료된 사용자: ", redis_key)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- 평면 배열 형식을 Lua Key-Value 테이블로 변환
local session_data = {}
for i = 1, #res, 2 do
    session_data[res[i]] = res[i + 1]
end
-- =========================================================================
-- 4. 보안 정책 검증 (Token Consistency Check)
-- 클라이언트 토큰과 Redis 저장 토큰 일치 여부 비교 (중복 로그인 방지)
-- =========================================================================
if session_data.token ~= pure_token then
    ngx.log(ngx.ERR, "토큰 불일치 (중복 로그인 등에 의해 갱신됨): ", member_id)
    ngx.exit(ngx.HTTP_UNAUTHORIZED)
end
-- =========================================================================
-- 5. 게이트웨이 통과 및 헤더 주입
-- 인증된 정보를 마이크로서비스(Downstream) 헤더에 주입
-- =========================================================================
ngx.req.clear_header("X-User-Id")
ngx.req.clear_header("X-Role")
ngx.req.set_header("X-User-Id", tostring(member_id))
ngx.req.set_header("X-Role", tostring(session_data.role))
-- Redis 커넥션 풀 반환
red:set_keepalive(10000, 100)

public / board 스키마: 유저 기본 정보와 커뮤니티 도메인을 물리적으로 분리.

social_account: OAuth 연동 정보 관리. Cascade 설정을 통해 회원 탈퇴 시 연쇄 삭제 및 무결성 유지.

    password VARCHAR(255) NOT NULL,                                 -- 암호화된 비밀번호
    name VARCHAR(255) NOT NULL,                                     -- 사용자 실명 또는 닉네임
    address VARCHAR(255),                                           -- 배송 및 기본 주소 정보
    age VARCHAR(255),                                               -- 연령대 (통계 및 필터링용)
    phone VARCHAR(255),                                             -- 연락처
    role VARCHAR(255) DEFAULT 'USER',                               -- 사용자 권한 (USER, ARTIST, ADMIN 등)
    info JSONB,                                                     -- 기타 메타데이터 (유연한 확장을 위한 JSON)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP   -- 계정 생성 일시

-- 2. 소셜계정(social_account) 테이블: OAuth 연동 정보 관리
CREATE TABLE public.social_account (

    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,  -- 소셜 연동 완료 일시

               REFERENCES public.member(member_id) ON DELETE CASCADE, -- 회원 탈퇴 시 연쇄 삭제
    CONSTRAINT uidx_provider_provider_id UNIQUE (provider, provider_id) -- 동일 제공자 내 중복 가입 방지
);
-- 특정 회원의 소셜 연동 정보 조회 성능 최적화를 위한 인덱스
CREATE INDEX idx_social_account_member_id ON public.social_account(member_id);

CREATE SCHEMA IF NOT EXISTS board;
-- 3. 게시판(board) 테이블: 커뮤니티 게시글 및 집계 데이터 관리
CREATE TABLE board.board (
    board_id BIGINT PRIMARY KEY GENERATED BY DEFAULT AS IDENTITY,  -- 게시글 고유 식별자
    category VARCHAR(20) NOT NULL,                                 -- 게시판 카테고리 (NOTICE, FREE 등)
    title VARCHAR(255) NOT NULL,                                   -- 게시글 제목
    content TEXT NOT NULL,                                         -- 게시글 본문 내용
    member_id BIGINT NOT NULL,                                     -- 작성자 member_id (Core 서비스 참조)
    like_count INTEGER DEFAULT 0 CHECK (like_count >= 0),          -- 추천수 (음수 방지 제약)
    comment_count INTEGER DEFAULT 0 CHECK (comment_count >= 0),    -- 댓글수 (음수 방지 제약)
    view_count INTEGER DEFAULT 0 CHECK (view_count >= 0),          -- 조회수 (음수 방지 제약)
    is_artist_post BOOLEAN DEFAULT FALSE,                          -- 아티스트(공식) 작성 글 여부 플래그

    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP  -- 마지막 수정 시각
);
-- 카테고리별 최신글 페이징 조회 성능을 높이기 위한 복합 인덱스
CREATE INDEX idx_board_category_created_at ON board.board(category, created_at DESC);
-- 좋아요 10개 초과인 인기글 조회를 최적화하기 위한 부분 인덱스(Partial Index)
CREATE INDEX idx_board_likes ON board.board(like_count DESC) WHERE like_count > 10;
-- 데이터 일관성 유지를 위한 자동 업데이트 트리거 함수
CREATE OR REPLACE FUNCTION board.update_modified_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
-- board 테이블에 트리거 적용
CREATE TRIGGER update_board_modtime

2. Payment (결제) 시스템
결제(Pay) 시스템은 데이터 정합성, 동시성 제어(Double Spending 방지), 외부 API 장애 격리가 가장 중요.
이를 위해 스키마를 분리하고, 원장(Ledger) 형태의 트랜잭션 테이블을 구성하는 것이 최신 MSA 트렌드.
pay 스키마: public 스키마 대신 도메인별 스키마를 분리하여 관리.
wallets (지갑): 사용자의 현재 잔액과 상태를 관리.
charges (외부 충전): 외부 결제 API(PG)와의 통신 내역 및 상태를 추적. 장애 발생 시 대사 작업(Reconciliation)에 필수.
transactions (포인트 원장): 포인트의 증감(충전, 결제, 환불)이 일어나는 모든 내역을 불변(Immutable)의 형태로 기록.
-- msa_pay_db
CREATE SCHEMA IF NOT EXISTS pay;
-- 1. 지갑(wallets): 사용자의 자산 잔액 상태 관리
CREATE TABLE pay.wallets (
    wallet_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),               -- 분산 환경용 UUID 식별자
    member_id BIGINT NOT NULL UNIQUE,                                   -- Core 회원 ID와 1:1 매핑
    balance NUMERIC(15, 2) NOT NULL DEFAULT 0.00 CHECK (balance >= 0),  -- 현재 잔액 (소수점 2자리)
    status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE',                       -- 계좌 상태 (ACTIVE, FROZEN)
    version INTEGER NOT NULL DEFAULT 0,                                 -- 낙관적 락(Optimistic Lock) 버전
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,      -- 지갑 생성일
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP       -- 마지막 변경일
);
-- 2. 충전(charges): 외부 PG사 연동 내역 및 결과 추적
CREATE TABLE pay.charges (
    charge_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),             -- 충전 요청 고유 ID
    wallet_id UUID NOT NULL REFERENCES pay.wallets(wallet_id),        -- 충전 대상 지갑 ID
    pg_provider VARCHAR(50) NOT NULL,                                 -- PG사 이름 (Kakaopay, Toss 등)
    pg_transaction_id VARCHAR(255),                                   -- PG사 연동 거래 번호
    amount NUMERIC(15, 2) NOT NULL CHECK (amount > 0),                -- 충전 요청 금액 (양수만 허용)
    status VARCHAR(20) NOT NULL DEFAULT 'PENDING',                    -- 처리 상태 (SUCCESS, FAILED)
    error_message TEXT,                                               -- 실패 시 에러 메시지
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,    -- 요청 시각
    completed_at TIMESTAMP WITH TIME ZONE                             -- 완료 시각
);
-- 3. 원장(transactions): 모든 포인트 증감 내역을 기록하는 불변 데이터
CREATE TABLE pay.transactions (
    transaction_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),        -- 거래 기록 고유 ID
    wallet_id UUID NOT NULL REFERENCES pay.wallets(wallet_id),        -- 대상 지갑 ID
    transaction_type VARCHAR(20) NOT NULL,                            -- 거래 유형 (CHARGE, PAYMENT, REFUND)
    amount NUMERIC(15, 2) NOT NULL,                                   -- 증감액 (+/-)
    balance_after NUMERIC(15, 2) NOT NULL,                            -- 거래 직후 잔액 (검증용)
    reference_id VARCHAR(255) NOT NULL,                               -- 주문번호 등 외부 참조 ID
    description VARCHAR(255),                                         -- 거래 상세 설명
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP     -- 기록 시각
);
-- 성능 최적화 인덱스
CREATE INDEX idx_transactions_wallet_id ON pay.transactions(wallet_id);
CREATE UNIQUE INDEX uidx_transactions_reference ON pay.transactions(reference_id, transaction_type);
2. Shop (커머스) 시스템
상품 재고 보존 및 주문 시점의 가격 변동을 방어하는 데이터 스냅샷 유지가 핵심.
shop 스키마: 커머스 도메인을 분리하고 상태값 관리를 위한 전용 ENUM 활용.
products: 상품 정보 및 재고 관리. Soft Delete(is_active) 적용으로 주문 내역과의 참조 무결성 확보.
orders: 주문 흐름 및 결제 상태 관리. 전체 프로세스의 마스터 원장 역할.
order_items: 주문 상세 내역 기록. 상품 가격 변동에 영향받지 않도록 주문 당시 가격(unit_price)을 기록.
unit_price: 이 설계에서 가장 중요한 컬럼.  products 테이블의 가격은 변하더라도, order_items에 '주문 시점의 가격을 복사'해두어야 고객의 실제 결제 금액과 주문 내역이 일치
영문 상태값 (English Status)
한국어 번역 (Korean Translation)
비즈니스 의미 (Business Meaning)
PENDING
결제 대기
주문서가 생성되었으나 아직 결제가 완료되지 않은 상태
PAID
결제 완료
결제가 확인되어 판매자가 배송을 준비해야 하는 상태
SHIPPED
배송 중
상품이 출고되어 택배사로 전달된 상태
COMPLETED
주문 완료
배송이 끝나고 구매 확정까지 이루어진 최종 상태
CANCELLED
주문 취소
결제 전 취소 혹은 환불 처리가 완료된 상태
-- msa_shop_db
-- shop 서비스 스키마 생성
CREATE SCHEMA IF NOT EXISTS shop;
-- 0. 상태 및 유형 관리를 위한 ENUM 정의
CREATE TYPE shop.seller_type AS ENUM ('ARTIST', 'USER');             -- 판매자 구분 (아티스트/일반유저)
CREATE TYPE shop.product_category AS ENUM ('OFFICIAL', 'UNOFFICIAL', 'SECONDHAND'); -- 상품 카테고리
CREATE TYPE shop.order_status AS ENUM ('PENDING', 'PAID', 'SHIPPED', 'COMPLETED', 'CANCELLED'); -- 주문 진행 상태
-- 1. 상품(products): 여러 옵션이 공유하는 상품의 기본/공통 정보
CREATE TABLE shop.products (
    product_id  UUID PRIMARY KEY DEFAULT gen_random_uuid(),         -- 상품 고유 ID
    seller_id   BIGINT NOT NULL,                                    -- 판매자 식별자 (Member 서비스 참조)
    seller_type shop.seller_type NOT NULL,                          -- 판매자 유형 (아티스트, 일반유저)
    category    shop.product_category NOT NULL,                     -- 상품 분류
    title       VARCHAR(255) NOT NULL,                              -- 상품명
    description TEXT,                                               -- 상품 상세 설명
    image_url   VARCHAR(500),                                       -- 상품 대표 이미지 경로 또는 URL 저장
    base_price  NUMERIC(15, 2) NOT NULL CHECK (base_price >= 0),    -- 상품 기본 가격
    is_active   BOOLEAN DEFAULT TRUE,                               -- 판매 활성화 여부 (Soft Delete 대용)
    created_at  TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP, -- 등록 일시
    updated_at  TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP  -- 수정 일시
);
-- 2. 상품 상세 옵션(product_variants): 재고 관리 및 실제 판매가 이루어지는 최소 단위 (SKU)
CREATE TABLE shop.product_variants (
    variant_id       UUID PRIMARY KEY DEFAULT gen_random_uuid(),         -- 옵션 고유 ID
    product_id       UUID NOT NULL REFERENCES shop.products(product_id) ON DELETE CASCADE, -- 부모 상품 ID
    color            VARCHAR(50),                                        -- 색상 옵션
    size             VARCHAR(50),                                        -- 사이즈 옵션
    additional_price NUMERIC(15, 2) DEFAULT 0,                           -- 기본가에 추가되는 금액
    stock_quantity   INT NOT NULL DEFAULT 0 CHECK (stock_quantity >= 0), -- 해당 옵션의 현재 재고 수량
    sku_code         VARCHAR(100) UNIQUE,                                -- 자체 관리용 재고 코드
    created_at       TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at       TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 3. 주문(orders): 결제 및 배송 처리를 위한 주문 요약 정보
CREATE TABLE shop.orders (
    order_id         UUID PRIMARY KEY DEFAULT gen_random_uuid(),         -- 주문 고유 ID
    member_id        BIGINT NOT NULL,                                    -- 구매자 식별자 (Member 서비스 참조)
    total_amount     NUMERIC(15, 2) NOT NULL CHECK (total_amount >= 0),  -- 총 결제 금액
    status           shop.order_status DEFAULT 'PENDING',                -- 주문/결제/배송 상태
    shipping_address TEXT,                                               -- 배송 목적지 주소
    created_at       TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at       TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 4. 주문 상세(order_items): 주문 당시의 상품 옵션 및 가격 정보 기록 (스냅샷)
CREATE TABLE shop.order_items (
    order_item_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),            -- 상세 품목 고유 ID
    order_id      UUID NOT NULL REFERENCES shop.orders(order_id) ON DELETE CASCADE, -- 부모 주문 ID
    variant_id    UUID NOT NULL REFERENCES shop.product_variants(variant_id), -- 구매한 상품 옵션 ID
    quantity      INT NOT NULL CHECK (quantity > 0),                     -- 구매 수량
    unit_price    NUMERIC(15, 2) NOT NULL CHECK (unit_price >= 0),       -- 구매 시점의 개당 단가 (옵션가 포함)
    created_at    TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 5. 장바구니(cart_items): 사용자가 구매 전 보관하는 상품 목록
CREATE TABLE shop.cart_items (
    cart_item_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),             -- 장바구니 항목 ID
    member_id    BIGINT NOT NULL,                                        -- 사용자 식별자
    variant_id   UUID NOT NULL REFERENCES shop.product_variants(variant_id) ON DELETE CASCADE, -- 담은 상품 옵션 ID
    quantity     INT NOT NULL DEFAULT 1 CHECK (quantity > 0),            -- 담은 수량
    created_at   TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at   TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uidx_cart_member_variant UNIQUE (member_id, variant_id)    -- 유저별 동일 옵션 중복 방지
);
-- 6. 주요 검색 및 조회를 위한 인덱스 설정
CREATE INDEX idx_products_category ON shop.products(category);           -- 카테고리별 검색 최적화
CREATE INDEX idx_products_seller ON shop.products(seller_id, seller_type); -- 판매자별 상품 목록 조회
CREATE INDEX idx_products_active ON shop.products(is_active) WHERE is_active = TRUE; -- 판매 중 상품 필터링 최적화
CREATE INDEX idx_product_variants_product ON shop.product_variants(product_id); -- 상품 상세 옵션 조회 성능 향상
CREATE INDEX idx_orders_member ON shop.orders(member_id);                -- 유저별 주문 내역 조회
CREATE INDEX idx_orders_status ON shop.orders(status);                   -- 배송/결제 상태별 관리 조회
CREATE INDEX idx_order_items_order_id ON shop.order_items(order_id);     -- 주문별 상세 품목 조회
CREATE INDEX idx_cart_items_member ON shop.cart_items(member_id);        -- 유저별 장바구니 조회
-- 7. 시간 업데이트 자동화를 위한 트리거 함수 및 적용
CREATE OR REPLACE FUNCTION shop.update_modified_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP; -- 데이터 수정 시 updated_at을 현재 시간으로 갱신
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
-- 수정 일시 자동 갱신 트리거 등록
CREATE TRIGGER update_products_modtime BEFORE UPDATE ON shop.products FOR EACH ROW EXECUTE PROCEDURE shop.update_modified_column();
CREATE TRIGGER update_product_variants_modtime BEFORE UPDATE ON shop.product_variants FOR EACH ROW EXECUTE PROCEDURE shop.update_modified_column();
CREATE TRIGGER update_orders_modtime BEFORE UPDATE ON shop.orders FOR EACH ROW EXECUTE PROCEDURE shop.update_modified_column();
CREATE TRIGGER update_cart_items_modtime BEFORE UPDATE ON shop.cart_items FOR EACH ROW EXECUTE PROCEDURE shop.update_modified_column();
3. Reservation (예약) 시스템
고트래픽 선착순 처리 및 서비스 간 낮은 결합도(Loose Coupling) 유지가 핵심.
res 스키마: 예약 도메인 장애 격리 및 독립성 보장.
events: 공연 정보 및 잔여석 관리. version 컬럼을 이용한 낙관적 락(Optimistic Lock)으로 동시성 제어.
reservations: 예약 확정 내역 관리. FK 제약조건을 생략하여 Core 시스템과의 물리적 의존성 제거.
-- msa_res_db
CREATE SCHEMA IF NOT EXISTS res;
-- 1. 이벤트(events): 공연 정보 및 실시간 잔여석 관리
CREATE TABLE res.events (
    event_id SERIAL PRIMARY KEY,                               -- 이벤트 고유 번호
    title VARCHAR(255) NOT NULL,                               -- 공연 제목
    artist_name VARCHAR(100),                                  -- 출연 아티스트
    event_type VARCHAR(50),                                    -- 공연 성격 (CONCERT, FANMEETING)
    price INTEGER NOT NULL CHECK (price >= 0),                 -- 티켓 단가
    total_capacity INT NOT NULL CHECK (total_capacity > 0),    -- 총 좌석 수
    available_seats INT NOT NULL CHECK (available_seats >= 0), -- 현재 남은 좌석 수
    event_date TIMESTAMP WITH TIME ZONE NOT NULL,              -- 공연 일시
    open_time TIMESTAMP WITH TIME ZONE NOT NULL,               -- 티켓 오픈 시각
    close_time TIMESTAMP WITH TIME ZONE NOT NULL,              -- 티켓 마감 시각
    version INTEGER NOT NULL DEFAULT 0,                        -- 낙관적 락용 버전 (선착순 처리 핵심)
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 2. 위치 정보(event_locations): 상세 장소 및 지도 데이터 (1:1)
CREATE TABLE res.event_locations (
    event_id INTEGER PRIMARY KEY REFERENCES res.events(event_id) ON DELETE CASCADE,
    venue VARCHAR(255) NOT NULL,                               -- 공연 장소 (기존 events에서 분리)
    address VARCHAR(500) NOT NULL,                             -- 장소 상세 주소 (기존 events에서 분리)
    latitude DECIMAL(10, 8),                                   -- 카카오/구글 지도 API 위도
    longitude DECIMAL(11, 8),                                  -- 카카오/구글 지도 API 경도
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 3. 사진 정보(event_images): 포스터 및 상세 페이지 사진 (1:N)
CREATE TABLE res.event_images (
    image_id SERIAL PRIMARY KEY,                               -- 이미지 고유 번호
    event_id INTEGER NOT NULL REFERENCES res.events(event_id) ON DELETE CASCADE,
    image_url VARCHAR(500) NOT NULL,                           -- S3 경로 등
    image_role VARCHAR(20) DEFAULT 'DETAIL'                    -- 포스터, 상세, 좌석배치도 등 역할 구분
        CHECK (image_role IN ('POSTER', 'DETAIL', 'LAYOUT')),
    sort_order INTEGER DEFAULT 0,                              -- 정렬 순서
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 4. 예약(reservations): 예매 확정 정보 및 티켓 코드
CREATE TABLE res.reservations (
    reservation_id UUID PRIMARY KEY DEFAULT gen_random_uuid(), -- 예약 고유 번호
    event_id INTEGER NOT NULL REFERENCES res.events(event_id), -- 대상 이벤트 ID
    member_id BIGINT NOT NULL,                                 -- 예매자 ID (외부 참조 - MSA 낮은 결합도를 위해 FK 생략)
    ticket_count INT NOT NULL CHECK (ticket_count > 0),        -- 매수
    total_price INT NOT NULL CHECK (total_price >= 0),         -- 총 결제액
    ticket_code VARCHAR(50) UNIQUE NOT NULL,                   -- 사용자용 티켓 식별 코드
    -- 허용되지 않은 상태값 삽입 방지 제약조건 추가
    status VARCHAR(20) DEFAULT 'PENDING'                       -- 상태값: 대기, 확정, 취소(변심), 환불(완료), 실패(시스템오류)
        CHECK (status IN ('PENDING', 'CONFIRMED', 'CANCELLED', 'REFUNDED', 'FAILED')),
    booked_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP -- 예매 완료 시각
);
---
-- 성능 최적화 인덱스
---
-- 검색 성능 최적화 (이벤트 기간 조회용)
CREATE INDEX idx_events_open_close ON res.events(open_time, close_time);
-- 특정 유저의 예매 내역 조회 최적화
CREATE INDEX idx_reservations_member ON res.reservations(member_id);
-- 검색 성능 최적화를 위한 인덱스 (특정 공연의 예매 현황)
CREATE INDEX idx_reservations_event ON res.reservations(event_id);
-- 공연별 이미지 리스트 조회 최적화
CREATE INDEX idx_event_images_event_id ON res.event_images(event_id);
4. Chat (채팅) 시스템
대량의 메시지 쓰기 성능 확보 및 실시간 조회를 위한 인덱스 최적화가 핵심.
chat 스키마: 실시간 통신 전용 스키마로 분리하여 I/O 부하 격리.
rooms: 채팅방 생성 및 그룹 여부 등 메타데이터 관리.
participants: 참여자 매핑 정보. last_read_message_id를 통한 안 읽은 메시지 동기화.
messages: 실제 메시지 데이터 기록. 성능 극대화를 위해 FK를 생략하고 시간순 조회를 위한 복합 인덱스 적용.
-- 채팅 스키마 생성
CREATE SCHEMA IF NOT EXISTS chat;
-- 1. 채팅방 테이블
CREATE TABLE chat.rooms (
    room_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    room_name VARCHAR(255),
    is_group BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
-- 2. 채팅방 참여자 테이블 (Redis에서 받은 member_id 사용)
CREATE TABLE chat.participants (
    room_id UUID NOT NULL,
    member_id BIGINT NOT NULL, -- Core 서비스의 BIGINT 타입과 통일
    joined_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    last_read_message_id BIGINT, -- 읽음 처리(안 읽은 메시지 수)를 위한 마커
    PRIMARY KEY (room_id, member_id)
);
CREATE INDEX idx_chat_participants_member ON chat.participants(member_id);
-- 3. 메시지 테이블
CREATE TABLE chat.messages (
    message_id BIGSERIAL PRIMARY KEY,
    room_id UUID NOT NULL,
    sender_id BIGINT NOT NULL, -- 작성자 (member_id)
    content TEXT NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
    -- 외래키는 생략 (MSA 결합도 최소화 및 쓰기 성능 향상 목적)
);
-- 채팅방 진입 시 과거 메시지를 빠르게 불러오기 위한 핵심 인덱스
CREATE INDEX idx_chat_messages_room_time ON chat.messages(room_id, created_at DESC);

각 라이브러리의 역할
Express
용도: Node.js를 위한 가장 대중적인 웹 프레임워크.
기능: HTTP 요청(GET, POST 등) 처리, 라우팅(경로 설정), 미들웨어 관리 등을 쉽게 할 수 있는 도구.
한 줄 요약: 웹 서버를 만들기 위한 도구.
amqplib
용도: RabbitMQ와 같은 메시지 브로커와 통신하기 위한 라이브러리.
기능: AMQP 0-9-1 프로토콜을 사용하여 메시지를 보내거나(Publish) 받는(Consume) 기능을 수행.
한 줄 요약: 메시지 큐(RabbitMQ)에 접속하기 위한 클라이언트 도구.
